# 为什么很多 fallback 设计最后都成了事故制造机

工程师对 fallback 往往天然有好感。

主路径失败了，还有备选路径。  
缓存挂了，还能直连数据库。  
服务 A 不行了，还能走服务 B。  
接口拿不到实时结果，还能拿一个兜底结果。

这套想法听起来很稳，甚至很符合常识。

但分布式系统里，很多 fallback 之所以危险，恰恰是因为它太符合直觉了。  
大家会本能觉得“多一层兜底总比没有好”，却低估了这层兜底在真实事故里的副作用。

## 先说结论

fallback 最大的问题，不是它一定没用。  
而是它会引入一条平时几乎不走、出事时才突然承担关键职责的新路径。

而这种路径通常同时具备四个特点：

- 难测试
- 不常演练
- 容易年久失修
- 一旦触发，可能把局部问题放大成系统性事故

所以在很多分布式系统里，fallback 不是保险。  
它更像一块平时看不见、只有出事时才踩到的薄冰。

## 为什么 fallback 往往比主路径更危险

### 1. 它很难在真实故障条件下被验证

这是第一个经常被低估的问题。

你当然可以写单元测试，也可以做一些故障注入。  
但 fallback 最麻烦的地方在于，它不是只在“某个函数返回错误”时才触发。

它真正危险的场景往往需要同时满足：

- 某个依赖失效
- 某类负载正在升高
- 某些缓存或局部状态刚好不在
- 多台机器在相近时间进入同一故障模式

这种组合条件，在测试环境里很难完整复现。  
于是 fallback 在纸面上看起来可用，在真正事故里却很可能第一次大规模被执行。

### 2. 它自己也会失败

很多人设计 fallback 时，下意识把它当成“主路径失败后的稳定保底”。

但现实是，fallback 只是另一条路径。  
既然是路径，它就同样可能：

- 超时
- 过载
- 数据不一致
- 返回低质量结果
- 把请求引到另一个脆弱依赖

也就是说，fallback 不是保证成功。  
它只是改变失败方式。

### 3. 它常常带着隐含 trade-off

如果 fallback 在所有维度上都更好，那它就不该叫 fallback，应该直接成为主路径。

既然它没有成为主路径，就说明它大概率在某些方面更差，比如：

- 更慢
- 更贵
- 更不一致
- 更脆弱
- 更容易扩大故障面

问题在于，很多设计只看到“还能跑”，却没认真评估这些 trade-off 会不会在事故里反过来伤害系统。

### 4. 它经常带来潜伏很久的 latent bug

fallback 最难受的一点，是它出 bug 时往往不是当天就暴露。

它可能几个月、几年都不触发。  
直到某个罕见条件刚好满足，这段代码才第一次在生产上承担关键流量。

到那时：

- 写这段逻辑的人可能已经不记得细节
- 依赖关系已经变了
- 主路径演化过很多次，fallback 没同步跟上
- 监控和报警也未必覆盖到了它

所以 fallback 很容易变成一种“静默积累复杂度”的装置。

## Amazon 那个经典案例，为什么值得反复读

AWS Builders' Library 里提到过一个很典型的事故。

当时 Amazon 网站为了展示商品配送时效，在网页服务器前加了一层本地缓存进程。  
主路径是：

- 网页服务器先查缓存
- 缓存命中后返回配送信息

这本来是为了保护后面的数据库，因为数据库承受不起所有页面直接查询。

问题出在 fallback：

- 如果缓存进程不健康
- 网页服务器就直接查数据库

这个逻辑平时看起来很自然。  
缓存挂了，总得继续给用户返回数据。

但真正的事故是：  
缓存后来在多个节点上几乎同时失败，于是所有网页服务器一起绕过缓存，直接打数据库。

结果不是“配送时效这个功能暂时不可用”。  
而是数据库被瞬间压死，整个网站页面都被拖下水，甚至进一步影响到后端履约系统。

这个案例最值得记住的，不是它有多戏剧性。  
而是它非常诚实地说明了一件事：

一个看起来局部、善意、合理的 fallback，完全可能把部分功能故障扩大成全站故障。

## 很多时候，明确失败比模糊兜底更安全

这句话一开始听上去有点反直觉。

但如果你的 fallback 会把问题扩大，那“继续给一个看起来没完全失败的结果”并不一定比“明确失败”更好。

因为明确失败至少有几个好处：

- 调用方知道应该按失败路径处理
- 不会偷偷把流量倒向一个更脆弱的依赖
- 不会在用户看来“看起来成功，实际上结果不可信”
- 系统的错误边界更清楚

所以很多场景里，更稳妥的策略不是 fallback，而是：

- fail fast
- 明确降级
- 返回受控的低功能模式

这和“偷偷走另一条风险更大的主路径替身”不是一回事。

## 比 fallback 更靠谱的几条路

### 1. 先把主路径做得更可靠

这是最不性感，但通常最有效的办法。

与其维护两条都会老化的路径，不如把工程投入优先砸在：

- 主依赖的可用性
- 数据本地化
- 缓存和预热
- 隔离和限流
- 更简单的调用图

很多时候，提高主路径成功率，带来的收益会比维护一条低频 fallback 更大。

### 2. 让调用方按统一规则处理失败

如果失败应该由上游来决定怎么处理，那么就别在中间层偷偷做一条复杂 fallback。

更稳的做法通常是：

- 返回清晰错误
- 让调用方按 timeout / retry / backoff / 幂等规则处理
- 让失败语义在调用链里保持一致

这样做虽然不华丽，但边界更清楚。

### 3. 尽量把“请求时取数据”变成“提前把数据放到本地”

这是 AWS 文里提到的另一个很重要的方向。

如果某类数据在请求到来前就已经被推到本地、预热到缓存，或者预计算好了，  
那么请求时就少了一层依赖，也就少了一次“要不要 fallback”的选择。

很多可靠性问题，最终不是靠更聪明的兜底解决的。  
而是靠减少实时链路上的移动部件解决的。

### 4. 如果替代路径真的必要，就别把它当冷备

如果业务上确实必须保留替代路径，那最差的做法就是让它长期躺着不动，只在事故时第一次上场。

更稳妥的办法是把它做成经常被验证、经常被演练、甚至持续被部分流量命中的路径。  
一旦走到这一步，它更接近 failover，而不是传统意义上的 fallback。

只有常跑、常测、常被监控的替代路径，才配在事故时承担责任。

## 一个更靠谱的判断问题

以后再有人提议“这里加个 fallback 吧”，我建议先追问四个问题：

1. 如果这条路径更可靠，为什么它不是主路径？
2. 它在真实故障条件下如何证明自己可用？
3. 它会不会把流量导向另一个更危险的依赖？
4. 如果它必须存在，我们准备如何持续演练它？

这四个问题里，只要有两个答不上来，这个 fallback 大概率就不值得轻易上线。

## 收束

很多系统不是缺兜底，而是缺边界感。

不是所有“还有另一条路可走”都叫稳。  
有些路只是平时不走，所以看起来像安全；一旦真走上去，才发现那是另一块更薄的冰。

所以如果你问我，系统设计里哪类“看起来更保险”的方案最需要怀疑，我会选 fallback。

因为它最容易把工程师的善意，变成事故里的额外故障面。
