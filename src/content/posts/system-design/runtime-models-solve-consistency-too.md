# Cloudflare Durable Objects 给系统设计的一条反直觉启发

一提到一致性问题，很多工程师的第一反应都很像：

是不是要更强的数据库？  
是不是要分布式锁？  
是不是要更复杂的事务协议？

这种反应很自然。  
因为一致性问题最常出现的地方，确实是在数据库、复制和并发写入上。

但 Cloudflare Durable Objects 这个案例给了一个很反直觉、也很值得记住的提醒：

有时候解决一致性问题，不一定要把数据库做得更复杂。  
你也可以先改运行时模型。

## 先说结论

并发正确性问题不只发生在数据库层，也发生在“谁拥有状态、谁来操作状态”的更上游层。

如果你能把同一份状态明确交给一个单线程对象处理，  
那么很多原本需要靠锁、事务和复杂协调兜住的问题，  
会在模型层先消失一大块。

所以系统设计不只是组件选型。  
它还包括一个更本质的问题：

这份状态，到底归谁负责？

## 为什么很多一致性 bug 其实不是数据库 bug

Cloudflare 那篇文章一上来就点得很准：

分布式存储和数据库当然很难，  
但应用代码自己的时序 bug 也一点不轻。

很多系统里的错误并不是因为数据库坏了，  
而是因为应用层在并发访问同一份状态时，写出了非常容易出 race condition 的代码。

比如一个最简单的逻辑：

1. 读出当前计数器
2. 加一
3. 写回去

看起来没问题。  
但只要两次请求并发交错，  
就可能一起读到同一个旧值，再一起写回，从而返回重复结果。

这类问题最麻烦的地方在于：

- 平时流量小的时候不明显
- 压力上来时才偶发出现
- 很难稳定复现
- 上线初期看起来一切正常

所以很多团队会本能觉得“是不是数据库语义不够强”。  
但有时候真正的问题不是存储太弱，而是并发入口太多。

## Durable Objects 真正有意思的，不是存储 API，而是状态所有权模型

Durable Objects 最值得记住的一点，不是它提供了什么键值接口，  
而是它把一份状态绑定给了一个对象，而这个对象在同一时刻只在一个位置、一个线程里运行。

这件事为什么重要？

因为它改变了问题形态。

在传统数据库模型下：

- 很多客户端都可能同时访问同一份状态
- 于是同步、锁、事务和竞争都自然涌进来

而在 Durable Objects 这种模型下：

- 某份状态明确属于某一个对象
- 某个时刻只有这一个执行单元在操作它

于是很多“多个参与者同时抢同一份状态”的问题，  
直接在入口处就被压缩了。

这不是说底层持久化 suddenly 不难了。  
而是说你先把最难的一类竞争模式，从编程模型里拿掉了。

## 单线程不等于天然没问题，但它大幅缩小了问题面

Cloudflare 那篇文章里有个很好的例子：

即便 Durable Object 本身是单线程，  
只要你在读写存储时 `await` 了 I/O，  
两个请求依然可能在这些暂停点之间交错执行。

这说明单线程本身不是魔法。  
只要执行流程会在 I/O 处让出控制权，  
race condition 依旧可能重新出现。

但这件事反而更说明问题：

真正困难的不是“线程多不多”，  
而是“状态访问是不是被一个清楚的所有权边界管理着”。

Durable Objects 后来的改进，  
本质上是在继续强化这个模型：  
尽量让开发者直觉上的写法，就更接近正确写法。

这特别值得系统设计学习时记下来。

因为很多时候，  
系统质量不只是来自更强的底层协议，  
还来自更不容易写错的运行时抽象。

## 为什么这对系统设计很重要

因为它提醒你，一致性问题有两种非常不同的解决方向。

### 1. 在共享竞争模型里继续加强控制

也就是大家更熟悉的路线：

- 锁
- 乐观并发控制
- 事务
- 共识
- 复制日志

这些当然重要。  
很多问题没有它们根本做不稳。

### 2. 直接改变“状态被谁拥有”的模型

另一条路线则是：

- 缩小状态共享面
- 让每份状态只有明确 owner
- 尽量让同一份状态只在一个执行上下文里被操作

这条路不一定总能走通，  
但一旦走通，它常常比继续堆协调机制更简单。

所以 Durable Objects 给系统设计最反直觉的一点，不是“又多了个新组件”，  
而是它提醒你：

有时更好的问题不是“怎么把共享状态协调好”，  
而是“为什么它一开始会被这么多人共享”。

## 一个很常见的误区：把一致性问题全都下沉给数据库

很多团队会默认：

应用层尽量简单，  
一致性让数据库或者中间件负责就好。

这句话有一部分对。  
但它也很容易把系统设计带进一个思维惯性：

只要出并发问题，就继续往下加更强的基础设施。

可现实里，基础设施越强，往往越贵、越重、越难操作。  
而如果上游状态边界本身就混乱，  
你只是把复杂度往底层推，并没有真的减少复杂度。

所以一致性设计不能只问：

- 数据库能不能扛

还得问：

- 状态所有权是不是已经清楚
- 是否真的需要这么多参与者同时碰这份状态

## 一个更靠谱的提问方式

以后再遇到一致性和并发正确性问题，  
我会建议先问下面这 4 句：

1. 这份状态到底由谁拥有？
2. 是不是很多执行单元在同时操作它？
3. 能不能把它收敛到更清楚的单一 owner？
4. 如果不能，再考虑事务、锁或更强协调是否值得？

只要先把这 4 个问题问完，  
很多“是不是该上更复杂数据库协议”的讨论会更清楚。

## 收束

Cloudflare Durable Objects 这条线最有启发性的地方，  
不是它证明了数据库不重要。  
恰恰相反，它说明了另一件更深的事：

一致性问题，很多时候在数据库之前就已经开始了。

所以如果你问我，这个案例最该给系统设计补的一课是什么，  
我会选这句：

“运行时模型也是一致性设计的一部分。”

因为系统设计不只是决定数据存哪、怎么复制。  
它还决定了这份状态一开始会不会被太多人一起碰。
